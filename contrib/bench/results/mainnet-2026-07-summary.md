# Mainnet benchmark summary — 2026-07-13

Reproduction of the repo's benchmarks against a real, fully synced mainnet
zebrad. Two parts:

- **Part A** — the contrib/bench hot read-path harness (ADR 0017 methodology),
  datasets extracted from the live mainnet node, run for both profiles.
- **Part B** — ingest throughput A/B/C: NEW Rust (this tree) vs OLD Rust
  baseline (pre-windowed-ingestor) vs Go reference, each ingesting from the
  real mainnet zebrad over fixed 8-minute windows.

## Part A — hot read-path harness (contrib/bench)

Workflow followed as documented in contrib/bench/README.md: `extract-dataset.sh
{dense,light}` against the live mainnet node, `populate.sh` per profile
(fairness verified: **12000/12000 blocks served from cache by both proxies,
GetBlockRange streams identical** for both profiles), `run-bench.sh` per
profile with the documented default parameter curve (REPS=5, DURATION=8s,
WARMUP=3s, CONCURRENCIES 1..64, all four request types), then
`aggregate.py`.

Datasets (extracted 2026-07-13):
- dense: [3350000, 3361999], avg 6 txids/block, avg 17,905 raw bytes/block
- light: [20000, 31999], avg 4.18 txids/block, avg 19,529 raw bytes/block

Raw ghz outputs, sidecar CPU samples, peak-RSS, footprints and final /metrics
scrapes live under `contrib/bench/results/{dense,light}/{rust,go}/`
(git-ignored). The tables below are the verbatim `aggregate.py` output.

Part A headline: Rust dominates GetBlockRange throughput once concurrency > 2
(dense: ~330k blocks/s plateau vs Go's 60-140k; light: ~105-146k vs 7-80k) and
holds flat GetBlock p99s under concurrency, where Go's p99 blows out to
33-55 ms at c>=16. Go wins low-concurrency dense range streaming (c=1-2) and
GetBlock p50 at moderate concurrency. Rust's redb cache is larger on disk
(2.3x both profiles); Go's peak RSS is ~3x Rust's on dense.

## Part B — ingest throughput vs the real mainnet node

Method: each implementation, one at a time (never concurrent), fresh empty
data dir, ingesting from the live mainnet zebrad for a fixed 480 s window;
blocks counted per the per-impl method (NEW: last `ingested ... to=` log;
OLD: last `ingesting height=` log, every-100 granularity; Go:
`db/main/lengths` size / 4). Binaries: NEW/OLD `cargo build --release`; Go
built with the one-line `NewBlockCache(..., <start>, ...)` patch (one binary
per range start; patch reverted immediately, `git status` clean).

| range | start | NEW Rust blocks (b/s) | OLD Rust blocks (b/s) | Go blocks (b/s) |
|---|---|---|---|---|
| R1 modern pre-spam | 1,500,000 | **238,912 (497.7)** | 26,401 (55.0) | 169,016 (352.1) |
| R2 sandblasting | 1,780,000 | **17,856 (37.2)** | 2,701 (5.6) | 3,170 (6.6) |
| R3 recent | 3,300,000 | 111,353 (232.0)* | 26,701 (55.6) | 111,367 (232.0)* |

\* R3 is **tip-capped for NEW Rust and Go**: both fully caught up to the chain
tip (~3,411,35x) inside the window and then idled. NEW Rust reached tip
~375 s in (effective catch-up ≈ 297 blocks/s); Go reached tip ~396 s in
(effective ≈ 281 blocks/s). The OLD baseline is nowhere near tip-capped.

Takeaways:
- NEW vs OLD Rust: 9.0x (R1), 6.6x (R2), 4.2x+ (R3, capped) — the windowed
  concurrent ingestor with batched commits is the dominant factor.
- NEW Rust vs Go: 1.41x on R1, 5.6x on R2 (sandblasting-era heavy shielded
  blocks), parity on R3 only because both were tip-capped.
- Errors: **none**. No txid-mismatch or hash-mismatch errors in any of the 9
  runs (grep over both server logs and Go's app.log). No reorg warnings.

Raw Part B logs/summaries were kept in the run’s scratch directory,
files `<impl>-<start>.{log,summary,errors}` plus `driver.log`. Data dirs were
deleted after each run per plan (sizes before deletion: new R1 2.9G, new R2
2.5G, new R3 241M, old R1 92M, old R2 356M, go R1 449M, go R2 297M).

## Deviations from the documented workflow, and caveats

1. **Host, not macOS**: harness was developed for Docker Desktop on macOS
   arm64; this run is native Docker on Debian amd64. No VM overhead, so
   absolute Part A numbers are better than the README's environment; the
   2 vCPU / 2 GiB per-proxy limits and all fidelity controls were unchanged.
2. **Portability fix (committed to the working tree)**:
   `scripts/extract-dataset.sh` — this host lacks `xxd` (and no root to
   install it); added a `hex2bin` fallback using `perl -0777 -ne 'print
   pack("H*", $_)'` when `xxd` is absent, and dropped `xxd` from the required
   tool check. No other harness script needed changes.
3. **jq**: host lacked `jq`; installed a static jq-1.7.1 binary into
   a user-local bin directory (user-level, no root). Not a repo change.
4. **populate.sh readiness heuristic fired early** for the Rust proxy on both
   profiles: redb grows its file by doubling, so file size plateaus while
   blocks are still being written, and the "size stable for 3 polls" check
   declared done too soon. The DENY_GETBLOCK verification then correctly
   failed (cache miss). Resolution: waited until the Rust cache truly reached
   the dataset tip and re-ran populate.sh; the recorded runs are the fully
   verified ones (12000/12000, fairness identical, both profiles). The
   heuristic could use a real completeness probe (e.g. GetBlock at the tip
   with DENY set) — left as a note, not changed.
5. **Dense rust redb footprint** (45.8 MiB) carries some churn from the
   interrupted first populate attempts (retries/backoff while DENY was set);
   a pristine single-pass ingest may be somewhat smaller. Go's footprint is
   unaffected.
6. **Light profile density**: 4.18 txids/block, not the ~1 tx/block the env
   comment hopes for at BASE=20000. Kept the documented BASE/SPAN unchanged
   (instructions: no new parameters). Compact blocks in this range are still
   pre-Sapling (no shielded data), so the profile still exercises the
   near-floor serialization path.
7. **Part A used the documented default curve** (no trimming was needed):
   REPS=5, DURATION=8s, WARMUP=3s, CONCURRENCIES="1 2 4 8 16 32 64",
   REQUESTS="getblock range100 range1000 range10000". Both sweeps exited 0.
8. **R3 tip-capping** (Part B) as noted above — 8-minute windows are longer
   than the catch-up time from 3,300,000 for NEW Rust and Go.
9. **Background load**: two zebrad nodes (mainnet + testnet) were running and
   syncing/gossiping throughout every measurement, plus the mainnet node was
   actively serving the Part B ingestors. All comparisons are like-for-like
   under the same load, but absolute numbers include this noise.
10. **Go binaries for Part B** were built from the Go repo's current master
    (61fee32), not the fdf1af5 pin the Part A container uses; the ingest path
    between those commits is unchanged (the delta is darkside/test fixes).
